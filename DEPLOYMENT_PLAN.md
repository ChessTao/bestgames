# План публикации страницы на сайте

Проект можно публиковать как статическую страницу, если для работы достаточно
HTML, CSS, JavaScript и локальных файлов проекта.

## 1. Проверить точку входа (выполнено)

Основной файл для публикации уже называется `index.html`. Перед загрузкой нужно
убедиться, что именно он открывает актуальную версию приложения и подключает
нужные файлы:

- `Styles.css`
- `app-pgn.js`, `app-render.js`, `app-position.js` или другие используемые JS-файлы
- локальные библиотеки из `vendor/`, если они нужны

## 2. Привести проект в чистый вид (выполнено)

Перед публикацией стоит проверить:

- все HTML-файлы сохранены в UTF-8;
- в `index.html` нет поврежденной разметки и лишних закрывающих тегов;
- пути к CSS, JS и библиотекам относительные;
- внешние CDN-зависимости либо доступны, либо заменены локальными файлами;
- приложение открывается локально без ошибок в консоли браузера.

## 3. Собрать минимальный комплект файлов (выполнено)

Для обычного статического хостинга обычно нужны:

- `index.html`
- `Styles.css`
- все JS-файлы, которые реально подключены страницей
- папка `vendor/`, если используются локальные библиотеки
- данные партий, если они вынесены в отдельные файлы

Файлы вроде `server.js`, `smoke-check.js`, `SMOKE_CHECK.md` и служебные заметки
не обязательно загружать на публичный сайт, если они не нужны в продакшене.

## 4. Выбрать место публикации

Возможные варианты:

- загрузить файлы через FTP/SFTP в папку сайта;
- использовать файловый менеджер хостинга;
- опубликовать через GitHub Pages, Netlify или Vercel;
- разместить проект в подпапке существующего сайта, например:

```text
/public_html/best-games/
```

Тогда страница будет доступна по адресу вроде:

```text
https://example.com/best-games/
```

## 5. Загрузить файлы на хостинг

Если страница должна открываться из отдельной папки сайта, нужно положить туда
`index.html` и все связанные файлы с сохранением структуры каталогов.

Если страница должна быть главной страницей сайта, `index.html` нужно положить в
корень публичной директории сайта, например `public_html`.

## 6. Проверить сайт после загрузки

После публикации нужно открыть страницу в браузере и проверить:

- загружается ли доска;
- работают ли переходы по ходам;
- отображаются ли партии и варианты;
- нет ли ошибок `404` для CSS, JS и библиотек;
- нет ли ошибок JavaScript в консоли разработчика.

## Рекомендуемый итог

Оптимальная продакшен-версия: один актуальный `index.html`, один основной CSS,
только реально используемые JS-файлы, локальная копия шахматной библиотеки и
чистые относительные пути. После этого страницу можно публиковать почти на любом
статическом хостинге.
